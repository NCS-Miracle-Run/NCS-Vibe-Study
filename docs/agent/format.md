# 문서 작성 규칙

`노트/*.md` 파일은 개념 정리용 문서다. `노트/소프트웨어공학.md` 형식을 기준으로 작성한다.

## 기본 구조

```markdown
# 문서 제목

## 목차

- [대분류](#대분류)
  - [중분류](#중분류)

---

## 대분류

### 중분류

* 핵심 설명
* 추가 설명
```

## 제목 규칙

- 문서 제목은 `#` 1개 사용
- 목차 제목은 `## 목차` 사용
- 큰 단원은 `##` 사용
- 세부 개념은 `###` 사용
- `####` 이하 소분류는 사용하지 않음
- 제목 끝에 불필요한 공백을 넣지 않음

## 목차 규칙

- 문서 상단에 목차를 둠
- 목차는 실제 제목 순서와 같게 작성
- `##`, `###`까지만 목차에 포함
- 목차와 본문 사이에는 `---` 구분선 삽입
- 제목을 바꾸거나 이동하면 목차 링크도 같이 수정

## 본문 규칙

- 본문 설명은 `*` 목록 사용
- 한 항목에는 한 개념만 작성
- 개념은 `* 개념명`으로 먼저 작성하고, 설명은 들여쓰기한 하위 목록으로 작성
- 같은 들여쓰기 레벨의 목록 항목 사이에는 빈 줄 하나를 추가
- 짧은 정의만 필요한 경우에만 `* 개념명: 설명` 형식 사용 가능
- 약어는 처음 작성할 때 풀어쓴 명칭을 함께 작성
- 약어 표기는 `풀어쓴 명칭(약어)` 또는 `약어(풀어쓴 명칭)` 형식 사용
- 본문, 목록, 표에서 볼드 처리(`**굵게**`) 금지
- 암기 순서는 `A → B → C` 형식 사용
- 비교나 분류가 많아도 표를 사용하지 않고 목록으로 작성
- 간단한 보충 설명은 인용문(`>`) 사용

## 표 규칙

- 표 사용 금지
- 비교, 분류, 구성 요소 정리는 목록으로 작성

## 코드블록 규칙

- 흐름도, 절차, 명령, 코드 예시는 코드블록 사용
- 언어가 명확하면 코드블록에 언어명 표시
- 단순 흐름도는 언어명 없이 작성 가능

## 구분선 규칙

- 큰 단원(`##`) 사이에는 필요하면 `---` 사용
- 같은 단원 안에서 너무 자주 사용하지 않음
- 목차와 본문 사이에는 반드시 `---` 사용

## 작성 예시

```markdown
## 요구사항 정의

### 요구사항 유형

* 기능 요구사항
  * 시스템이 제공해야 하는 기능

* 비기능 요구사항
  * 성능, 보안, 사용성 등 품질 조건

### 요구사항 개발 프로세스

* 도출(Elicitation) → 분석(Analysis) → 명세(Specification) → 확인(Validation)
```
